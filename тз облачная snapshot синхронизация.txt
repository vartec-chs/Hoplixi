Нужно спроектировать архитектуру облачной snapshot-синхронизации для локального менеджера паролей на Dart.

Цель
Создать безопасную и достаточно простую систему синхронизации через облачные провайдеры без merge содержимого базы данных. Синхронизация должна работать по модели snapshot, поддерживать несколько независимых хранилищ пользователя, минимизировать повторную перекачку attachment-файлов и быть пригодной для реализации на Dart.

Поддерживаемые облачные провайдеры
enum CloudSyncProvider {
  dropbox,
  google,
  onedrive,
  yandex
}

Общий контекст
Приложение хранит несколько независимых пользовательских хранилищ. Каждое хранилище синхронизируется как отдельная сущность. Основное содержимое хранилища:
- зашифрованная база данных;
- файл ключевой конфигурации;
- набор зашифрованных attachment-файлов;
- манифесты для синхронизации.

Структура одного локального хранилища
store_name/
  store_key.json
  store_manifest.json
  store_name.hplxdb
  attachments/
  attachments_decrypted/

Назначение файлов и директорий
1. store_key.json
   Содержит параметры для генерации ключа шифрования.
   Минимально:
   - salt
   - useDeviceKey

2. store_manifest.json
   Основной манифест конкретного хранилища.
   Должен содержать идентификаторы, ревизию, timestamp, данные о последнем snapshot, информацию о текущем provider для sync, хэши содержимого и служебные поля, необходимые для сравнения локальной и удалённой версии.

3. store_name.hplxdb
   Основная зашифрованная БД хранилища.

4. attachments/
   Папка с зашифрованными вложениями.
   Каждый attachment хранится отдельным файлом.
   Имена файлов имеют вид:
   <uuid.v4>.enc

5. attachments_decrypted/
   Временная локальная папка с расшифрованными файлами.
   Не должна участвовать в синхронизации.
   Не должна попадать в облако.
   Не должна учитываться при построении hash/manifest для sync.

Важные ограничения
1. Нельзя использовать merge содержимого БД.
   БД синхронизируется только как единый snapshot-файл.

2. Не нужно делать инкрементальную синхронизацию БД.
   Нужно делать snapshot-based sync.

3. Для attachments нельзя использовать только один общий hash папки как единственный механизм сравнения, потому что при изменении одного файла не должно требоваться скачивание всех attachment-файлов заново.

4. В облаке нельзя раскрывать исходные имена файлов и лишние метаданные о них.
   Исходные имена, MIME-тип, описание и другая бизнес-метаинформация лежат только в БД.
   В облаке для attachment-файлов разрешено хранить только технические данные, достаточные для синхронизации.

5. У пользователя может быть несколько независимых хранилищ в одном облачном аккаунте, поэтому нужно продумать общий индекс всех хранилищ.

Требуемая архитектура синхронизации

Нужно спроектировать три уровня данных:

1. cloud_manifest.json
   Общий облачный индекс всех хранилищ пользователя.
   Это не главный источник истины для каждого store, а индекс/реестр.
   Он нужен, чтобы:
   - быстро получить список всех удалённых хранилищ;
   - не обходить всё облако вручную при каждом старте sync;
   - хранить краткую сводку по каждому store;
   - понимать, где расположен каждый store.

2. store_manifest.json
   Главный манифест конкретного хранилища.
   Это источник истины для одного store.
   Должен использоваться для определения:
   - какая версия новее;
   - есть ли конфликт;
   - к какому store относится snapshot;
   - кем и когда был создан snapshot;
   - совпадает ли локальное и удалённое содержимое.

3. attachments_manifest.json
   Отдельный реестр attachment-файлов.
   Нужен для пофайловой синхронизации attachments.
   Он должен позволять:
   - понять, какие attachment-файлы нужно скачать;
   - какие уже есть локально;
   - какие удалены;
   - какие повреждены или не совпадают по хэшу;
   - не скачивать все attachments заново при изменении одного файла.

Рекомендуемая облачная структура
Нужно спроектировать и предложить финальный layout наподобие:

cloud_root/
  cloud_manifest.json
  stores/
    <storeUuid>/
      store_manifest.json
      attachments_manifest.json
      store_key.json
      store_name.hplxdb
      attachments/
        <uuid>.enc
        <uuid>.enc

Важно:
- в качестве имени папки store использовать storeUuid, а не storeName;
- storeName считать только отображаемым именем;
- attachments_decrypted никогда не хранить в облаке;
- можно предложить provider-specific вариант хранения пути или remote id, но общая логика должна оставаться одинаковой.

Версионирование snapshot
Нужно спроектировать надёжную схему версионирования snapshot без использования только одного timestamp.

Требования:
1. Для каждого store должен быть revision:
   - целочисленный, монотонно возрастающий;
   - увеличивается при каждом изменении содержимого snapshot;
   - является основным критерием определения более новой версии.

2. Должен быть updatedAt:
   - timestamp в UTC;
   - нужен как вспомогательное поле;
   - не должен быть единственным критерием определения новизны.

3. Должен быть snapshotId:
   - уникальный идентификатор конкретного snapshot;
   - может использоваться как tie-breaker или для журналирования.

4. Должен быть storeUuid:
   - постоянный уникальный идентификатор хранилища;
   - нужен для проверки, что сравниваются версии одного и того же store.

5. Желательно предусмотреть baseRevision и/или baseSnapshotId:
   - для optimistic concurrency;
   - чтобы понимать, на основе какой удалённой версии был создан новый локальный snapshot.

Правила сравнения версий
Нужно явно описать алгоритм compare для store_manifest:

1. Если storeUuid различается:
   - это разные хранилища, сравнивать их как версии одного store нельзя.

2. Если revision у одной стороны больше:
   - эта версия считается новее.

3. Если revision одинаковый и content hashes совпадают:
   - это одинаковое состояние.

4. Если revision одинаковый, но content hashes различаются:
   - это конфликт;
   - нельзя автоматически считать одну из версий “новее” только по updatedAt.

5. updatedAt может использоваться только как вспомогательное поле для диагностики, UI или tie-breaker в логах, но не как основной механизм разрешения конфликта при одинаковом revision и разном содержимом.

Требования к store_manifest.json
Нужно спроектировать финальную JSON-схему store_manifest.json.

Минимально нужно предусмотреть:
- manifestVersion
- storeUuid
- storeName
- revision
- updatedAt
- snapshotId
- baseRevision или baseSnapshotId
- lastModifiedBy:
  - deviceId
  - clientInstanceId
  - appVersion
- sync:
  - provider
  - remoteStoreId или remotePath
  - syncedAt
  - при необходимости providerRevision / etag / versionTag
- content:
  - dbFile:
    - fileName
    - size
    - sha256
  - keyFile:
    - sha256
  - attachments:
    - count
    - totalSize
    - manifestSha256
    - filesHash

Нужно объяснить назначение каждого поля.

Требования к cloud_manifest.json
Нужно спроектировать JSON-схему cloud_manifest.json как общего облачного индекса всех stores.

Он должен содержать:
- version
- updatedAt
- список stores

Для каждого store в списке:
- storeUuid
- storeName
- revision
- updatedAt
- snapshotId
- provider path / remoteStoreId
- возможно краткие summary hashes
- при необходимости deleted / archived флаг

Важно:
- cloud_manifest.json — это индекс, а не единственный источник истины;
- если cloud_manifest.json и store_manifest.json расходятся, приоритет у store_manifest.json конкретного store.

Требования к attachments_manifest.json
Нужно спроектировать JSON-схему attachments_manifest.json как реестра attachment-файлов.

Требования:
1. Реестр должен описывать все файлы из attachments/.
2. В нём не должно быть исходных имён файлов и других бизнес-метаданных.
3. Для каждого attachment достаточно хранить:
   - id
   - size
   - sha256
   При необходимости имя файла может формироваться как <id>.enc.
4. Нужен агрегированный filesHash.
5. Желательно хранить:
   - version
   - storeUuid
   - revision
   - updatedAt

Нужно описать, как считается filesHash.
Рекомендуемая схема:
- для каждой записи формируется строка:
  <id>:<size>:<sha256>
- записи сортируются по id;
- строки объединяются через \n;
- от результата вычисляется SHA-256;
- результат сохраняется в filesHash.

Важно:
- filesHash нужен для быстрой проверки, изменились ли attachments вообще;
- сам список files нужен для точечного сравнения и выборочной загрузки файлов.

Алгоритмы синхронизации
Нужно подробно спроектировать алгоритмы:

1. compare local/remote store_manifest
   Результат сравнения должен позволять понять:
   - identical
   - localNewer
   - remoteNewer
   - conflict
   - differentStores
   - invalidRemote / invalidLocal

2. compare local/remote attachments_manifest
   Результат должен содержать:
   - файлы, которые надо скачать;
   - файлы, которые надо загрузить;
   - файлы, которые надо удалить локально;
   - файлы, которые надо удалить в облаке;
   - файлы, которые совпадают и могут быть пропущены;
   - несовпадающие записи.

3. upload flow для store
   Нужен безопасный порядок:
   - загрузить store_name.hplxdb;
   - загрузить/обновить attachment-файлы;
   - загрузить attachments_manifest.json;
   - загрузить store_key.json при необходимости;
   - последним загрузить store_manifest.json как commit marker;
   - затем обновить cloud_manifest.json.

4. download flow для store
   Нужен безопасный порядок:
   - скачать store_manifest.json;
   - проверить storeUuid, revision, hashes;
   - скачать store_name.hplxdb;
   - скачать attachments_manifest.json;
   - сравнить attachment-реестры;
   - скачать только недостающие/изменённые attachment-файлы;
   - удалить локально отсутствующие в remote attachment-файлы;
   - проверить хэши;
   - обновить локальные manifest-файлы.

5. Обработка частично завершённого upload/download
   Нужно продумать, как система должна вести себя, если:
   - БД загружена, а store_manifest ещё нет;
   - attachments частично загружены;
   - attachments_manifest обновлён, а часть файлов не успела загрузиться;
   - cloud_manifest устарел, а store_manifest уже новый.

Ключевая идея:
- store_manifest.json должен играть роль commit marker для конкретного store;
- cloud_manifest.json должен быть только индексом и обновляться уже после успешного commit store.

Требования к конфликтам
Нужно предложить простую и безопасную политику конфликтов.

Минимально:
1. Если local.revision == remote.revision, но хэши содержимого различаются:
   - это конфликт.

2. Если local.baseRevision < remote.revision и локально есть несинхронизированные изменения:
   - это потенциальный конфликт или требуется ручной выбор.

3. Нельзя merge БД автоматически.

4. Нужно предложить варианты поведения приложения:
   - остановить sync и показать конфликт пользователю;
   - предложить сделать force upload локальной версии;
   - предложить скачать remote как отдельную копию;
   - предложить создать backup локального состояния перед заменой.

Безопасность и приватность
Нужно учитывать следующие принципы:

1. attachments в облаке не должны раскрывать исходные имена файлов.
2. В manifest-файлах не должно быть лишних пользовательских метаданных.
3. Можно хранить только технические поля, необходимые для sync.
4. attachments_decrypted не должны участвовать в sync.
5. storeName может считаться менее чувствительным полем, но при желании нужно предложить вариант, где даже оно не используется в облачном пути, а только storeUuid.
6. Нужно минимизировать утечки метаданных через manifest-файлы.

Оптимизация
Нужно предусмотреть практичные оптимизации:

1. Не скачивать все attachments при изменении одного файла.
2. Сравнивать attachments по реестру, а не по одному общему hash.
3. Можно предложить локальный кэш file hash для ускорения пересчёта attachments_manifest:
   - например, по комбинации path/id, size, mtime, sha256;
   - но облачный manifest всё равно должен опираться на полный sha256.
4. Если filesHash совпадает, можно не сравнивать каждый файл повторно, кроме случаев принудительной проверки.
