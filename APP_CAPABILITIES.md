# Hoplixi: что уже есть в приложении

Этот файл собран по [AGENT.md](AGENT.md), [README.md](README.md) и текущему коду. Он нужен как краткая карта уже существующего функционала приложения.

## 1. Что такое Hoplixi

Hoplixi это кроссплатформенный менеджер паролей и защищённое хранилище данных.

Поддерживаемые платформы:

- Android
- iOS
- Windows
- Linux
- macOS

## 2. Ключевые пользовательские возможности

### Менеджер паролей и защищённое хранилище

- создание и открытие локальных хранилищ
- блокировка и разблокировка хранилища
- хранение разных типов защищённых записей в одном vault
- генерация паролей
- OTP / 2FA
- история изменений записей
- архивирование хранилища
- импорт данных и миграции

### Безопасность

- шифрование основной базы через SQLite3 Multiple Ciphers
- отдельное шифрование файлов и вложений
- защищённое хранение ключей и чувствительных настроек
- биометрическая аутентификация через `LocalAuthService`
- PIN-код и настройки авто-блокировки
- write policy для чувствительных настроек через биометрию

### LocalSend

В приложении есть отдельный модуль локального обмена данными между устройствами.

Что умеет:

- поиск устройств в локальной сети
- signaling-сессия между устройствами
- WebRTC-передача файлов
- передача текста
- отправка архива хранилища
- отправка зашифрованного пакета OAuth-токенов cloud sync
- подтверждение входящего подключения
- история отправок и получений
- обработка разрыва соединения и ошибок передачи

Основные части модуля находятся в `lib/features/local_send/`.

### Cloud Sync и snapshot sync

В проекте уже есть новый модуль облачной синхронизации, не только заготовка.

Что реализовано:

- OAuth-авторизация облачных провайдеров
- хранение OAuth-токенов
- refresh токенов и авторизованный HTTP transport
- единый storage-слой поверх облачных API
- подключение текущего хранилища к облачному аккаунту
- snapshot sync текущего store
- сравнение локальной и удалённой ревизии
- upload локального snapshot в облако
- download удалённого snapshot из облака
- обработка конфликтов версий
- ручное разрешение конфликта: оставить локальную или принять удалённую версию
- прогресс синхронизации для UI
- сигнал о необходимости ручной переавторизации
- синхронизация перед закрытием хранилища

Поддержанные провайдеры в новом модуле:

- Dropbox
- Google Drive
- OneDrive
- Yandex Disk

Snapshot sync работает через отдельную подсистему в `lib/features/cloud_sync/snapshot_sync/`.

## 3. Какие сущности поддерживает vault

Базовые vault-сущности:

- пароли
- OTP
- заметки
- банковские карты
- документы
- файлы
- контакты
- API-ключи
- SSH-ключи
- сертификаты
- криптокошельки
- Wi-Fi сети
- identity / личные данные
- лицензионные ключи
- recovery codes
- карты лояльности

Служебные и связанные сущности:

- категории
- теги
- иконки
- пользовательские поля
- метаданные файлов
- страницы документов
- ссылки заметок на другие vault-сущности
- история изменений для каждой основной сущности

Архитектурно все основные записи опираются на общую базовую таблицу `vault_items`, а типоспецифичные поля вынесены в отдельные таблицы.

## 4. Система фильтрации и организации данных

В дашборде реализована отдельная система фильтрации по типам сущностей.

Что уже есть:

- отдельные filter provider'ы почти для каждого типа сущности
- отдельные UI-секции фильтров почти для каждого типа сущности
- фильтрация по категориям
- фильтрация по тегам
- поиск по категориям и тегам
- пагинация категорий и тегов
- debounce для поиска
- хранение выбранных фильтров в состоянии drawer
- синхронизация фильтров с базовым `baseFilterProvider`

Дополнительно для организации данных есть:

- избранное
- архив
- pinned-записи
- soft delete
- учёт использования и recent score

## 5. Система истории и аудита

Для записей реализована полноценная история изменений:

- общая таблица истории `vault_item_history`
- отдельные history-таблицы для каждого типа сущности
- фиксация действий create / modify / delete
- хранение снимков полей на момент изменения
- экран просмотра истории
- поиск по истории
- восстановление данных из истории

## 6. Работа с файлами и документами

В приложении есть отдельная файловая подсистема.

Она включает:

- хранение зашифрованных вложений
- работу с файловыми метаданными
- хранение документов и страниц документов
- агрегированный текст документа
- OCR-текст страниц на уровне модели данных
- сервисы для файлового и документного хранилища
- импорт и передача архивов хранилища

## 7. Настройки и системные возможности

В приложении уже присутствуют:

- начальный setup flow
- настройки темы
- переключение языка
- launch at startup
- auto-lock timeout
- backup path и backup interval
- настройки cloud sync и backup-флаги
- системный трей на desktop
- глобальные горячие клавиши
- multi-window инфраструктура
- логирование, crash reports и viewer логов

## 8. Внутренние технические подсистемы

Ключевые внутренние блоки проекта:

- `main_store/` для зашифрованной Drift/SQLite базы
- `core/app_prefs/` для типобезопасных настроек
- `core/logger/` для логирования и crash-репортов
- `core/services/` для биометрии, secure storage и общих сервисов
- `routing/` на `go_router`
- `shared/ui/` для общих UI-компонентов
- `packages/file_crypto` и Rust crypto API для криптографических операций
- `packages/cloud_storage_sdk` для облачного слоя
- `packages/card_scanner` для сканирования карт
- `packages/secure_clipboard_win` для безопасного clipboard на Windows

## 9. Итог

На текущем этапе Hoplixi это уже не просто менеджер паролей, а платформа для защищённого локального хранилища с:

- множеством типов сущностей
- историей изменений
- системой категорий, тегов и фильтров
- биометрической защитой
- шифрованием базы и файлов
- локальным обменом данными через LocalSend
- облачным snapshot sync для store
